# Production Documentation - 本番環境用ドキュメント

このディレクトリには、実装に直接使用するドキュメントが含まれています。

---

## 📁 含まれるドキュメント

### 1. requirements.md (8.2KB) - 要件定義書
**最重要ドキュメント**

**内容:**
- プロジェクト概要
- 技術要件（HTML5, CSS3, Vanilla JavaScript）
- ページ1: 年齢確認ページ
- ページ2: オープニングアニメーション
- ページ3: メインページ
- デザイン要件（カラーパレット、フォント）
- 機能要件（チェックリスト付き）
- パフォーマンス要件
- テスト要件

**いつ使う:**
- プロジェクト開始時
- 機能仕様の確認時
- クライアントとの確認時
- テスト項目作成時

---

### 2. implementation-plan.md - 実装計画書
**実装の道しるべ**

**内容:**

- 実装フェーズ（Phase 1-8）
- 技術実装の詳細（コードサンプル付き）
- テストチェックリスト
- 今後の拡張予定

**いつ使う:**
- 実装開始前
- Issue作成時
- タスク優先順位決定時
- 進捗確認時

  - 動作フロー図
- CSSの骨組み

**いつ使う:**
- 実装の最初のステップ
- 動作確認のテスト時
- 基本構造の理解時

**いつ使う:**

## 📊 ドキュメント利用フロー

### ステップ1: プロジェクト理解
```
requirements.md を読む
   ↓
production-sitemap.md でサイト全体を把握
   ↓
sitemap-visual.md で視覚的に確認
```

### ステップ2: 実装計画
```
implementation-plan.md でフェーズを確認
   ↓
walking-skeleton-flow.md で最初のタスクを理解
   ↓
優先順位に従ってIssue作成
```

### ステップ3: 実装開始
```
walking-skeleton-flow.md のコードサンプルを参照
   ↓
production-sitemap.md でファイルパス確認
   ↓
requirements.md で機能要件を確認
```

### ステップ4: テスト
```
requirements.md のテスト要件を確認
   ↓
implementation-plan.md のチェックリストで検証
```

## 📝 ドキュメント更新ルール

### いつ更新するか
- 機能追加・変更時
- バグ修正で仕様が変わった時
- クライアントからの要望変更時

### 更新方法
1. 該当ドキュメントを開く
2. 変更箇所を編集
3. 「最終更新日」を更新
4. 関連ドキュメントも確認・更新

### バージョン管理
- 大きな変更の場合は、古いバージョンを残す
- ファイル名に日付を追加: `requirements-2025-11-06.md`

---

## ⚠️ 注意事項

1. **これらのドキュメントは常に最新に保つ**
   - 実装と乖離しないよう注意

2. **変更履歴を残す**
   - なぜ変更したかをコメントで記録

3. **チーム全員が参照できるようにする**
   - 場所を共有
   - 更新を通知

4. **本番環境にはデプロイしない**
   - `.gitignore` で除外（任意）
   - または別リポジトリで管理
